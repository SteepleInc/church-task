# E2E-first UI testing strategy

Church Work tests product UI behavior primarily through Playwright workflow tests against the real TanStack Start app, Testcontainers Postgres, Drizzle migrations/seeds, a real Zero process where required, and Better Auth. The product needs confidence in authenticated workflows such as onboarding and task creation more than confidence in isolated component rendering. Pure React component tests are not part of the default strategy; backend invariants, authorization rules, and data edge cases are tested separately with Vitest against the Postgres/Drizzle/Effect packages. E2E tests may use one real signup/login smoke path, but most authenticated workflow tests should start from backend-supported Better Auth session seeding and Drizzle seed/reset helpers.
